## 4. 데이터베이스 구성 및 역할

| DB             | 역할                                                          | 비고                     |
| -------------- | ------------------------------------------------------------- | ------------------------ |
| **PostgreSQL** | 메인 RDB — 사용자, 콘티, 찬양, 팀, 채팅, 히스토리, 커뮤니티 | Supabase 호스팅          |
| **Redis**      | Refresh Token, 이메일 인증 토큰                               | Upstash (TLS, rediss://) |

### 악보 저장 방식

- PDF/이미지 악보는 **Supabase Storage** 사용
- DB에는 파일 URL만 저장, 실제 파일은 클라우드 스토리지에 보관

### 로그인 설계

- 이메일/비밀번호 + **이메일 인증 필수**
- Redis로 JWT Refresh Token 및 이메일 인증 토큰 관리 (TTL)
- 소셜 로그인(Google 등) 추후 고려

## 7. 배포 방법

### 프론트엔드

- 도메인: `https://worshiplog.inho.pe.kr`

### 백엔드 — Oracle Cloud (Ubuntu)

- pm2로 프로세스 관리
- GitHub Actions SSH 자동 배포 (`server/**` 변경 시)
- `.env`는 서버에 직접 유지 (배포 스크립트가 일부 값만 주입)

**주의사항**:

- `CLIENT_URL`에 프로덕션 도메인 포함 필요 (CORS)
  - 예: `CLIENT_URL=https://worshiplog.inho.pe.kr,http://localhost:3001`
- `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET` 누락 시 로그인 500 에러
- `REDIS_URL` (Upstash rediss://) 형식으로 설정

## 8. 악보 저작권 처리 방안

- 악보는 **로그인 세션 내 개인 사용**으로 제한
- 다른 유저와의 직접 악보 공유 기능 비활성화
- **PDF 출력 시에만 악보 포함** → 출력물은 개인 오프라인 사용 목적
- 팀 내 공유 시에는 악보 파일이 아닌 **콘티 정보(곡명, 키, 템포 등)만 공유**

---

## 9. 팀 초대 기능 동작 방식

1. 팀장이 **초대 링크 또는 QR 코드 생성** (서버에서 토큰 발급, 만료 시간 설정)
2. 초대받은 유저가 링크/QR 접속 → 로그인 후 팀 자동 가입
3. 토큰은 1회용 + 시간 제한 (24시간)

**구현:**

- `invite_token` 발급 (UUID) → Redis TTL 저장
- QR 코드는 `qrcode` 라이브러리로 생성
- 이메일 초대: 대상 이메일로 초대 링크 발송

## 11. 커뮤니티 & 말씀 랜덤 추천

### 커뮤니티

- 로그인한 사용자만 접근 가능
- 카테고리 분류 (자유, 묵상나눔 등)
- 댓글 기능

### 말씀 랜덤 추천

- 성경 말씀은 **JSON 파일로 서버에 번들 저장**
  - 위치: `server/src/common/data/bible.json`
  - 구조: `[{ "book": "창", "chapter": 1, "verse": 1, "content": "..." }, ...]`
- 매일 또는 접속 시 랜덤 말씀 추천 → 관련 찬양 큐레이션 연결

## 12. DB 주의사항

- `prisma db push`는 Supabase pooler(6543)에서 실패
  → 직접 SQL 실행 후 `npx prisma generate`
- FK 컬럼 타입은 `UUID` 사용 (TEXT 사용 시 FK constraint 오류)
- RLS: 활성화 + policy 없이 설정 (Prisma 서버는 postgres 슈퍼유저로 RLS 우회)
